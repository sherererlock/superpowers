# 赏析：test-driven-development（测试驱动开发技能）

## 技能定位与核心目标

`test-driven-development` 是 Superpowers 体系中的**质量基石技能**，为所有功能实现和 bug 修复提供底层的测试纪律保障。核心目标是：通过严格的红-绿-重构循环，确保每一行生产代码都有对应的失败测试作为前提，从根本上杜绝"测试只是验证代码能跑"的错误认知。

这个技能不只是 TDD 方法论的说明书，更是一套**反理性化（anti-rationalization）框架**——它预见了所有可能的借口，并逐一封堵。

---

## 核心设计哲学

### 1. 铁律而非建议（The Iron Law）

技能的核心规则以"铁律"（Iron Law）命名：

```
NO PRODUCTION CODE WITHOUT A FAILING TEST FIRST
```

这不是最佳实践，而是不可违反的规则。更关键的是，技能对"违反后怎么办"给出了明确指令：**删除已写的代码，重新开始**。不能保留作"参考"，不能"边看边改"，不能"偷看一眼"——删除就是删除。

这种极端的规则设计背后有清醒的认知：一旦允许例外，例外就会成为常态。

### 2. 观察失败是强制步骤（Verify RED is Mandatory）

技能的核心原则之一："If you didn't watch the test fail, you don't know if it tests the right thing."（如果你没有看着测试失败，你就不知道它是否测试了正确的东西。）

"观察测试失败"被设计为**强制步骤**，不可跳过。这个设计的逻辑是：测试通过有两种可能——代码正确，或者测试本身有问题（测试了错误的东西、测试了已有行为、存在拼写错误）。只有先看到正确的失败，才能确认测试在测正确的事情。

### 3. 最小实现原则（Minimal Code）

GREEN 阶段的规则是"写最简单的代码让测试通过"，明确禁止：
- 添加未被测试要求的功能
- 重构其他代码
- "改进"超出测试范围的内容

这是对 AI 过度实现倾向的直接约束。技能用 YAGNI 原则的反例（带 `maxRetries`、`backoff`、`onRetry` 的过度设计函数）来说明什么是错误的，对比鲜明。

### 4. 五类反模式的系统性封堵

技能专门列出了五类 TDD 反模式，每一类都包含：错误示例、为什么错误、正确做法、以及一个"门控函数"（Gate Function）——一个在执行操作前必须回答的问题清单。

这种"门控函数"设计是技能中最精妙的部分：它将"应该怎么做"转化为"在做之前必须回答这些问题"，强制 AI 在行动前进行反思。

### 5. 违反字面规则即违反精神（Letter = Spirit）

技能明确声明："Violating the letter of the rules is violating the spirit of the rules."（违反规则的字面意思就是违反规则的精神。）

这是对"我理解了精神，所以可以灵活处理字面规则"这种常见理性化的预防性封堵。技能不允许 AI 以"理解了 TDD 的精神"为由跳过具体步骤。

## 关键机制细节

### 红-绿-重构循环的流程图设计

技能使用 DOT 语言绘制了完整的 TDD 循环流程图，包含了两个关键的反馈路径：
- **错误失败 → 重写测试**：如果测试失败的原因不对（如拼写错误而非功能缺失），回到 RED 阶段
- **重构后验证**：REFACTOR 阶段完成后必须重新验证 GREEN，确保重构没有破坏行为

### 好测试 vs 坏测试的对比

技能通过具体代码示例对比了好测试和坏测试：

**好测试**（重试操作示例）：
- 测试真实行为（实际调用了多少次）
- 名称描述行为（"retries failed operations 3 times"）
- 不依赖 mock 的调用次数

**坏测试**（同一场景）：
- 测试 mock 行为（`expect(mock).toHaveBeenCalledTimes(3)`）
- 名称模糊（"retry works"）
- 验证的是 mock 配置，而非真实代码逻辑

### 五类反模式详解

**反模式 1：断言 Mock 元素**
测试检查带有 `-mock` 后缀的测试 ID，意味着测试只在 mock 环境下有意义，对真实行为毫无保障。

**反模式 2：生产代码中的测试专用方法**
为了让测试能访问内部状态而在生产代码中添加方法，污染了生产代码的接口。正确做法是将这些方法移到测试工具文件中。

**反模式 3：不理解依赖就 Mock**
"为了安全而 mock"是最危险的反模式。门控函数要求在 mock 任何方法前，必须先回答：这个方法有什么副作用？测试依赖这些副作用吗？如果不确定，先用真实实现运行测试，观察实际需要什么，再做最小化 mock。

**反模式 4：不完整的 Mock**
只 mock 了"你认为需要的字段"，但下游代码可能依赖你没有包含的字段，导致测试通过但集成失败。铁律：mock 必须镜像真实 API 的完整数据结构。

**反模式 5：测试作为事后补充**
实现完成后才写测试，这根本不是 TDD。测试是实现的一部分，不是可选的后续步骤。

### 后写测试为何无效

技能专门解释了"先实现后写测试"的根本缺陷：后写的测试会立即通过，而立即通过证明不了任何事情——可能测试了错误的东西，可能测试了实现细节而非行为，可能遗漏了你在实现时忘记的边界情况。

### 何时 Mock 是正确的

技能没有一刀切地反对 mock，而是给出了清晰的边界：
- Mock 用于隔离，不是用来测试
- 在正确的层级 mock（mock 慢的/外部的操作，而非测试依赖的高层方法）
- 当 mock 设置比测试逻辑还长时，考虑改用集成测试

## 亮点总结

| 亮点 | 说明 |
|------|------|
| 铁律命名 | "Iron Law"强调不可妥协性，删除违规代码不留余地 |
| 观察失败强制化 | "Verify RED"作为不可跳过步骤，确认测试测的是正确的事 |
| 字面=精神声明 | 预防"理解精神可以灵活字面"的理性化 |
| 五类反模式系统封堵 | 每类反模式配备门控函数，强制行动前反思 |
| 门控函数设计 | 将规则转化为"行动前必须回答的问题"，更易执行 |
| 好坏测试代码对比 | 具体代码示例，而非抽象原则 |
| 不完整 Mock 铁律 | Mock 必须镜像完整真实结构，防止静默失败 |
| 后写测试的缺陷分析 | 解释为何后写测试从根本上无效 |
| Mock 复杂度警告 | Mock 设置>50%测试代码时，考虑集成测试 |
| 反理性化语言 | "Thinking 'skip TDD just this once'? Stop. That's rationalization." |

## 总结性评价

`test-driven-development` 是 Superpowers 体系中**规则密度最高**的技能。它的每一条规则都不是孤立的建议，而是针对一个具体的失败模式设计的防护措施。

这个技能最值得关注的设计是**门控函数**（Gate Function）模式：在执行某个操作（如添加 mock）之前，必须先回答一系列问题。这种设计将"应该怎么做"转化为了"做之前必须经过的检查点"，比单纯的规则陈述更难被绕过。

"违反字面规则即违反精神"这条元规则同样精妙——它封堵了 AI 最常见的理性化路径：声称理解了精神，然后以此为由跳过具体步骤。技能通过这条规则，将所有具体步骤都提升到了不可妥协的层级。

五类反模式的系统性梳理，揭示了 TDD 在实践中最容易出错的地方：不是不知道 TDD，而是在具体实现时不知不觉地滑向了反模式。将这些反模式显式化并配备门控函数，是这个技能最务实的贡献。

对于任何采用 AI 辅助开发的团队，这个技能提供的不只是 TDD 方法论，更是一套**防止 AI 在测试纪律上走捷径的制度性约束**。
