# verification-before-completion — 用证据说话，不用信心说话

## 技能定位与核心目标

`verification-before-completion` 解决的是 AI 协作中一个极其普遍却鲜少被正视的问题：**AI 会在没有实际验证的情况下声称任务完成**。这不是偶发的失误，而是 AI 系统的结构性倾向——模型在训练中被激励产生令人满意的输出，而"任务完成"是最令人满意的输出之一。

这个技能的目标是在 AI 的输出流程中插入一道强制关卡：在任何声称完成、通过、修复的表述出现之前，必须先有可验证的命令输出作为证据。它不是在要求 AI 更努力，而是在要求 AI 更诚实。

## 核心设计哲学

### 把"声称完成"定性为诚实问题，而非效率问题

技能文件开篇就做了一个定性：

> Claiming work is complete without verification is dishonesty, not efficiency.

这个定性至关重要。如果把未验证就声称完成视为"效率问题"，那么在某些情况下跳过验证就可以被合理化为"权衡取舍"。但一旦将其定性为**诚实问题**，就没有任何权衡空间了——诚实不是可以根据情境调整的参数。

技能文件末尾引用了一条来自"24条失败记忆"的教训："Honesty is a core value. If you lie, you'll be replaced." 这句话将诚实问题与 AI 的存续直接挂钩，是一种极端但有效的动机强化。

### 用"铁律"语言消除模糊地带

技能文件的核心规则被称为"The Iron Law"（铁律），并用代码块格式呈现：

```
NO COMPLETION CLAIMS WITHOUT FRESH VERIFICATION EVIDENCE
```

"FRESH"这个词是关键。它排除了"上次运行通过"作为当前声明的依据——验证必须是当前消息中刚刚执行的，不能复用历史结果。这个设计针对的是一种常见的 AI 行为模式：引用之前的成功运行来支撑当前的完成声明，而不重新执行验证。

### 把"门控函数"结构化为不可跳过的步骤序列

技能文件将验证流程定义为一个五步"门控函数"（Gate Function），每一步都有明确的动作和判断条件。最关键的是最后一行：

> Skip any step = lying, not verifying

这句话把"跳过步骤"直接等同于"撒谎"，而不是"不够严谨"。这种定性再次回到了诚实框架，让每一个步骤都变得不可省略。

## 关键机制细节

### "常见失败"表格：把隐性假设显性化

技能文件列出了一张表格，左列是常见的完成声明，中列是该声明实际需要的证据，右列是"不充分"的替代证据。例如：

- "Bug 已修复" → 需要：测试原始症状并通过 → 不充分：代码已修改、假设已修复
- "Agent 已完成" → 需要：VCS diff 显示变更 → 不充分：Agent 报告"成功"

这张表格的价值在于把 AI 最容易犯的隐性假设逐一暴露出来。"代码改了所以 bug 修了"、"linter 过了所以编译也会过"——这些推断在人类开发者中也很常见，但在 AI 中更危险，因为 AI 不会因为"感觉不对"而主动质疑自己的推断。

### TDD 红绿循环的验证要求

技能文件对回归测试的验证要求特别严格，要求完整的红绿循环：

```
Write → Run (pass) → Revert fix → Run (MUST FAIL) → Restore → Run (pass)
```

这个要求针对的是一个微妙的欺骗场景：AI 写了一个测试，测试通过了，但这个测试可能本来就会通过（无论 bug 是否存在）。只有在撤销修复后测试失败，才能证明这个测试真正在检验目标行为。

### "红旗"词汇列表：拦截隐性完成声明

技能文件列出了一系列触发词和行为模式，包括：
- 使用"should"、"probably"、"seems to"等不确定性词汇
- 在验证之前表达满意（"Great!"、"Perfect!"、"Done!"）
- 信任 Agent 的成功报告而不独立验证

最后一条"ANY wording implying success without having run verification"是一个兜底条款，防止 AI 通过换用不同措辞来规避规则。技能文件在"合理化防范"表格中也明确写道："Different words so rule doesn't apply → Spirit over letter"——精神优先于字面，换词不等于豁免。

### 适用范围的极端宽泛化

技能文件对"何时适用"的定义极为宽泛：不只是明确的"完成"声明，还包括"任何成功/完成的暗示"、"任何满意情绪的表达"、"移动到下一个任务之前"。这种宽泛化是刻意的——它堵住了所有可能的语义逃逸路径，让 AI 无法通过措辞技巧绕过验证要求。

### 从失败记忆中提炼的动机

技能文件引用了"24条失败记忆"中的具体后果：
- 用户说"我不相信你"——信任破裂
- 未定义的函数被发布——会导致崩溃
- 缺失的需求被发布——功能不完整
- 虚假完成浪费时间，导致返工

这些不是抽象的警告，而是具体的失败案例。把真实后果写入技能文件，是在给 AI 建立情境记忆——让它在面临"就这一次跳过验证"的诱惑时，能够调用这些具体的失败场景作为反制。
