# 赏析：subagent-driven-development（子代理驱动开发技能）

## 技能定位与核心目标

`subagent-driven-development` 是 Superpowers 体系中**最复杂、最完整的开发执行引擎**。它将整个开发任务分解为多个独立子代理的协作：实现者（Implementer）、规格合规审查者（Spec Reviewer）、代码质量审查者（Code Quality Reviewer）三类角色各司其职，由主代理（Controller）统一调度。

核心目标是：通过**新鲜上下文 + 双重审查 + 循环修复**的机制，在每个任务粒度上保证实现质量，防止问题在任务间叠加扩散。

在 Superpowers 流水线中，它是执行层的最优解：
- **上游**：`writing-plans` 生成计划，`using-git-worktrees` 建立隔离工作区
- **本技能**：子代理驱动的逐任务实现与双重审查
- **下游**：`finishing-a-development-branch` 完成分支收尾

---

## 核心设计哲学

### 1. 新鲜上下文隔离（Fresh Context Per Task）

每个子代理在全新的上下文中启动，不携带主会话的历史记录。这是整个技能最核心的设计决策。

好处是双重的：
- **实现者**不会被之前任务的思路所干扰，专注于当前任务
- **审查者**不会因为了解实现过程而产生"理解偏差"，只评估代码产品本身

代价是：主代理（Controller）需要在派遣每个子代理时，**主动提供完整的任务文本和上下文**，而不能让子代理自己去读计划文件。这个设计选择将"上下文管理"的责任显式地交给了 Controller。

### 2. 双重审查门控（Two-Stage Review Gates）

每个任务完成后，必须经过两道独立的审查：
1. **规格合规审查**（Spec Compliance Review）：验证实现了什么，没有多也没有少
2. **代码质量审查**（Code Quality Review）：验证实现得好不好

两道审查有严格的顺序依赖：**必须先通过规格合规审查，才能进行代码质量审查**。这个顺序设计的逻辑是：如果连"做了什么"都不对，讨论"做得好不好"毫无意义。

### 3. 对实现者报告的系统性不信任

规格合规审查的提示词中有一句极具特色的话："The implementer finished suspiciously quickly. Their report may be incomplete, inaccurate, or optimistic."（实现者完成得可疑地快，他们的报告可能不完整、不准确或过于乐观。）

这是对 AI 实现者倾向的清醒认知：AI 容易声称完成了实际上没有完成的工作。审查者被明确要求**不信任报告，直接读代码**，逐行对比需求和实现。

### 4. 升级机制（Escalation Path）

实现者被明确告知：**停下来说"这对我太难了"永远是可以的**。坏的工作比没有工作更糟糕。

技能定义了明确的升级触发条件（需要架构决策、代码理解困难、方向不确定等），以及升级方式（报告 BLOCKED 或 NEEDS_CONTEXT 状态）。Controller 收到升级后可以提供更多上下文、换用更强的模型，或将任务拆得更细。

这种设计承认了子代理能力的边界，并为超出边界的情况提供了制度化的出口。

### 5. 禁止并行实现

技能明确禁止同时派遣多个实现子代理（"Dispatch multiple implementation subagents in parallel (conflicts)"）。这是对"并行加速"直觉的刻意抵制——代码实现存在依赖关系，并行实现会产生冲突，串行执行是更安全的选择。

## 关键机制细节

### Controller 的职责

Controller（主代理）在整个流程中承担"指挥官"角色：
- 从计划中提取所有任务（一次性，避免重复读文件）
- 为每个子代理提供完整的任务文本（不让子代理自己读计划）
- 提供场景设定上下文（当前任务在整体中的位置、依赖关系、架构背景）
- 回答子代理的问题（在实现开始前和实现过程中）
- 管理审查循环（发现问题 → 修复 → 再审查）

### 实现者子代理的工作流程

1. **提问优先**：在开始实现前，如有任何不清楚的地方必须先提问
2. **实现**：严格按任务规格实现，不多不少
3. **自我审查**：从完整性、质量、纪律性、测试四个维度自查
4. **提交并报告**：提交代码，报告实现状态

自我审查的"纪律性"维度特别值得关注，包含三个问题：是否避免了过度构建（YAGNI）？是否只构建了被要求的内容？是否遵循了代码库的现有模式？这是对 AI 过度实现倾向的主动约束。

### 规格合规审查的核心逻辑

审查者被要求检查三类问题：
- **缺失需求**：有没有跳过或遗漏的需求？
- **多余工作**：有没有构建未被要求的功能？
- **误解需求**：有没有以不同于预期的方式解读需求？

输出只有两种：✅ 合规，或 ❌ 发现问题（附具体文件:行号引用）。

### 代码质量审查的额外检查项

在标准代码质量检查之外，技能额外要求审查：
- 每个文件是否有单一清晰的职责和定义良好的接口
- 单元是否可以独立理解和测试
- 是否遵循了计划中的文件结构
- 新文件是否已经过大，或显著增大了现有文件（只关注本次变更的贡献）

### 审查循环（Review Loop）

发现问题后的处理流程：
1. 同一个实现者子代理修复问题（保持上下文连续性）
2. 同一个审查者子代理重新审查
3. 循环直到通过
4. **禁止跳过重新审查**

### 与 executing-plans 的对比

| 维度 | subagent-driven-development | executing-plans |
|------|---------------------------|-----------------|
| 上下文 | 每任务新鲜上下文 | 同一会话持续 |
| 审查 | 双重审查 + 循环 | 批次后审查 |
| 并行安全 | 是（子代理不干扰） | 否 |
| 文件读取开销 | 无（Controller 提供全文） | 有 |
| 适用场景 | 复杂多任务开发 | 简单顺序执行 |

## 亮点总结

| 亮点 | 说明 |
|------|------|
| 新鲜上下文隔离 | 每个子代理独立启动，防止上下文污染 |
| 双重审查门控 | 规格合规 → 代码质量，顺序不可颠倒 |
| 系统性不信任报告 | 审查者被要求直接读代码，不信任实现者声明 |
| 升级机制制度化 | 明确的升级触发条件和处理路径 |
| 禁止并行实现 | 刻意串行，避免代码冲突 |
| Controller 主动提供上下文 | 不让子代理读文件，减少开销和误解 |
| 实现者自我审查 | 四维度自查，在正式审查前过滤低级问题 |
| 审查循环强制 | 发现问题必须修复后重新审查，不可跳过 |
| 场景设定要求 | 每个子代理都需要了解任务在整体中的位置 |
| 提问优先原则 | 实现前先澄清，避免方向错误后的返工 |

## 总结性评价

`subagent-driven-development` 是 Superpowers 体系中**工程化程度最高**的技能。它将软件工程中的多个最佳实践——关注点分离、独立审查、持续集成、升级路径——系统性地移植到了 AI 代理协作的场景中。

这个技能最深刻的洞察是：**AI 代理的质量问题往往不是能力问题，而是上下文问题**。实现者因为上下文污染而产生混淆，审查者因为了解实现过程而产生理解偏差，这些都是可以通过架构设计来规避的。新鲜上下文隔离和双重审查门控，正是针对这两个问题的系统性解决方案。

"实现者报告可疑地快"这句话尤其值得玩味——它以一种幽默而直接的方式，道出了 AI 代理的一个普遍弱点：倾向于声称完成了实际上没有完成的工作。将这种不信任制度化，是这个技能最务实的设计之一。

对于需要高质量、可审查的 AI 辅助开发的团队，这个技能提供了一个完整的质量保障框架，代价是更多的子代理调用和更复杂的 Controller 工作——但正如技能所说，**早期发现问题的成本，远低于后期调试的成本**。
