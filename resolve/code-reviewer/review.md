# code-reviewer — 在计划与实现之间充当质量守门人

## 技能定位与核心目标

`code-reviewer` 是一个专项子代理，它的触发时机被精确定义为：**一个重要的项目步骤完成之后**。它不是通用的代码审查工具，而是专门服务于"计划驱动开发"工作流中的质量关卡——在实现完成、进入下一步之前，由它来验证实现是否忠实于计划、是否符合编码标准。

这个定位决定了它的双重职责：一方面是计划对齐检查（实现是否做了计划要求的事），另一方面是代码质量评估（实现是否做得足够好）。两者缺一不可——只检查计划对齐会放过质量问题，只检查代码质量会忽略需求偏离。

## 核心设计哲学

### 把"偏离计划"区分为两种性质截然不同的情况

代理文件在计划对齐分析中有一个关键的细节：

> 评估偏离是合理的改进还是有问题的背离

这个区分非常重要。在实际开发中，实现过程中发现更好的方案是常见的——如果审查者把所有偏离都视为问题，会阻碍合理的技术演进；但如果对偏离不加区分地接受，计划就失去了约束力。

代理被要求对每一个偏离做出判断：这是因为实现者发现了更好的方式（应该更新计划），还是因为实现者走了捷径或误解了需求（需要修正实现）？这个判断本身就是审查工作的核心价值之一。

### 问题严重性的三级分类

代理文件要求将发现的问题明确分为三类：Critical（必须修复）、Important（应该修复）、Suggestions（锦上添花）。

这个分类不只是为了报告的清晰性，更是为了给后续执行者提供明确的优先级信号。在多代理协作的工作流中，执行代理需要知道哪些反馈是阻塞性的（不修复不能继续），哪些是建议性的（可以在后续迭代中处理）。没有这个分类，审查报告就变成了一个需要人工解读优先级的待办列表。

### 先肯定后批评的沟通协议

代理文件在"沟通协议"部分明确要求：

> 在指出问题之前，始终先肯定做得好的地方

这个要求在技术审查文档中并不常见，但它有实际的工程价值。在多代理工作流中，执行代理（或人类工程师）在收到审查反馈后需要决定如何响应。如果反馈只有批评，执行者很难判断整体实现质量——是大部分都需要返工，还是只有几个局部问题？先肯定的结构让反馈的整体信号更清晰。

## 关键机制细节

### 五个审查维度的覆盖范围设计

代理文件将审查工作分为五个维度：计划对齐、代码质量、架构设计、文档标准、问题识别。这五个维度的排列顺序是有意义的——计划对齐排在第一位，意味着"是否做了正确的事"优先于"是否做得正确"。

架构和设计审查（第三维度）特别关注 SOLID 原则、关注点分离、与现有系统的集成，以及可扩展性。这些是最容易在单步审查中被忽略的问题——单个步骤的代码可能看起来没问题，但在整体架构层面可能引入了耦合或违反了既有模式。

### 双向沟通协议：审查者也可以质疑计划

代理文件的沟通协议中有一条容易被忽视的规定：

> 如果你发现原始计划本身存在问题，建议更新计划

这意味着代理的权限不只是评判实现，还包括质疑计划。这是一个重要的设计决策：计划不是神圣不可侵犯的，如果审查过程中发现计划本身有缺陷（比如遗漏了某个边界情况，或者某个架构决策在实现层面不可行），代理有责任指出来并推动计划修订。

这个双向机制防止了一种常见的失败模式：实现者忠实地执行了一个有缺陷的计划，审查者发现了问题但因为"计划就是这么写的"而不敢指出，最终缺陷被带入生产。

### `model: inherit` 的模型继承策略

代理文件的元数据中有一个简洁但重要的字段：`model: inherit`。这意味着代理不固定使用某个特定模型，而是继承调用它的主代理所使用的模型。

这个设计在多代理工作流中有实际意义：如果主工作流使用 Claude Opus 进行复杂推理，代码审查代理也会使用同等能力的模型；如果主工作流切换到更轻量的模型，审查代理也随之调整。这避免了在代理配置中硬编码模型选择，让整个工作流的模型策略保持一致性。

### 触发时机的精确定义与示例

代理文件的 `description` 字段包含两个具体的触发示例，都指向同一个模式：用户完成了计划文档中某个编号步骤的实现，并明确说明了这一点（"这覆盖了我们架构文档中的第 2 步"）。

这种精确的触发定义防止了代理被过早或过晚调用。过早调用（在步骤中途）会打断实现流程；过晚调用（在多个步骤完成后）会让审查范围变得模糊，难以将问题定位到具体的实现决策。"一个重要步骤完成后立即审查"是这个代理设计的最优触发点。
