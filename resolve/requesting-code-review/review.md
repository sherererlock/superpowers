# 赏析：requesting-code-review（请求代码审查技能）

## 技能定位与核心目标

`requesting-code-review` 是 Superpowers 体系中的**代码审查调度器**，负责在开发过程中的关键节点主动触发代码审查。它的核心目标是：通过派遣专门的 `code-reviewer` 子代理，在问题扩散之前捕获它们——而不是等到最后才做一次性的大规模审查。

这个技能与上一篇赏析的 `receiving-code-review` 形成了完整的闭环：一个负责"发出审查请求"，一个负责"接收审查反馈"。两者共同构成了 Superpowers 体系中的代码质量保障层。

---

## 核心设计哲学

### 1. 早审查、频审查（Review Early, Review Often）

技能的核心原则是"Review early, review often"。这与传统的"完成后统一审查"模式形成鲜明对比。

在 `subagent-driven-development` 工作流中，技能要求**每完成一个任务就触发一次审查**，而不是等所有任务完成后再审查。这种设计的逻辑是：早期发现的问题修复成本远低于后期发现的问题，尤其是当后续任务建立在有缺陷的基础上时，问题会以指数级方式扩散。

### 2. 上下文隔离（Context Isolation）

技能的一个关键设计决策是：审查者子代理获得的是**精心构建的上下文**，而不是当前会话的历史记录。

这背后有两层考量：
- **审查质量**：审查者应该评估代码产品本身，而不是被开发过程中的思考历程所影响
- **上下文保护**：将审查任务分离到子代理，避免消耗主代理的上下文窗口，让主代理可以继续后续工作

这是一种典型的"关注点分离"设计——开发上下文和审查上下文互不干扰。

### 3. 结构化的审查模板

技能提供了一个完整的 `code-reviewer.md` 模板，涵盖代码质量、架构、测试、需求符合度、生产就绪性五个维度，并要求按严重程度（Critical/Important/Minor）分类输出问题。

这种结构化设计有两个好处：
- **一致性**：每次审查的输出格式相同，便于比较和追踪
- **可操作性**：分级的问题清单让开发者知道优先处理什么

### 4. 强制性审查节点

技能区分了"强制审查"和"可选审查"场景：
- **强制**：子代理驱动开发中每个任务后、主要功能完成后、合并到主分支前
- **可选**：卡住时（获取新视角）、重构前（建立基线）、修复复杂 bug 后

这种区分避免了"审查疲劳"——不是所有时刻都需要审查，但关键节点不可跳过。

## 关键机制细节

### 审查请求的三步流程

**Step 1：获取 Git SHA**
```bash
BASE_SHA=$(git rev-parse HEAD~1)  # 或 origin/main
HEAD_SHA=$(git rev-parse HEAD)
```
通过精确的 commit 范围界定审查范围，避免审查者看到无关的历史变更。

**Step 2：派遣 code-reviewer 子代理**
填充模板中的五个占位符：
- `{WHAT_WAS_IMPLEMENTED}`：刚刚构建了什么
- `{PLAN_OR_REQUIREMENTS}`：应该做什么
- `{BASE_SHA}` / `{HEAD_SHA}`：审查的 commit 范围
- `{DESCRIPTION}`：简短摘要

**Step 3：处理反馈**
- Critical 问题：立即修复
- Important 问题：继续前必须修复
- Minor 问题：记录留待后续
- 审查者错误时：用技术理由推回

### code-reviewer 模板的审查维度

**代码质量层面：**
- 关注点分离是否清晰
- 错误处理是否完善
- 类型安全（如适用）
- DRY 原则
- 边界情况处理

**架构层面：**
- 设计决策是否合理
- 可扩展性考量
- 性能影响
- 安全隐患

**测试层面：**
- 测试是否真正测试逻辑（而非只测 mock）
- 边界情况覆盖
- 集成测试
- 所有测试是否通过

**需求符合度：**
- 计划中的所有需求是否满足
- 是否有范围蔓延
- 破坏性变更是否有文档

**生产就绪性：**
- 迁移策略（如有 schema 变更）
- 向后兼容性
- 文档完整性

### 输出格式的强制规范

模板明确规定了审查者的输出格式，并列出了"DO"和"DON'T"清单：

**必须做：**
- 按实际严重程度分类（不是所有问题都是 Critical）
- 具体到文件:行号
- 解释问题为什么重要
- 给出明确的合并建议

**禁止做：**
- 不检查就说"looks good"
- 把小问题标记为 Critical
- 对没有审查的代码给出反馈
- 给出模糊建议（如"改善错误处理"）
- 回避给出明确结论

### 与工作流的集成

- **子代理驱动开发**：每个任务后审查，防止问题叠加
- **执行计划**：每批次（3个任务）后审查
- **临时开发**：合并前或卡住时审查

## 亮点总结

| 亮点 | 说明 |
|------|------|
| 早审查频审查原则 | 每任务后触发，防止问题指数级扩散 |
| 上下文隔离设计 | 子代理获得精确上下文，不受开发历史干扰 |
| 主代理上下文保护 | 审查分离到子代理，主代理继续工作不受影响 |
| 五维度审查清单 | 代码质量/架构/测试/需求/生产就绪性全覆盖 |
| 三级严重程度分类 | Critical/Important/Minor，明确处理优先级 |
| 强制 vs 可选区分 | 避免审查疲劳，关键节点不可跳过 |
| 模板化输出规范 | 固定格式保证一致性和可操作性 |
| 推回机制保留 | 审查者错误时可用技术理由反驳 |
| 完整示例 | 提供端到端的使用示例，降低上手门槛 |

## 总结性评价

`requesting-code-review` 与 `receiving-code-review` 共同构成了 Superpowers 体系的**代码质量双保险**。前者负责主动触发审查并提供精确上下文，后者负责批判性地处理审查反馈。

这个技能最值得关注的设计是**上下文隔离**：通过将审查任务分离到独立子代理，既保证了审查的客观性（审查者不受开发过程影响），又保护了主代理的上下文窗口（开发可以继续进行）。这是一种在 AI 代理架构层面的精妙设计，而不仅仅是流程上的安排。

`code-reviewer.md` 模板的设计同样值得称道：它不只是一个检查清单，而是一个完整的审查框架，包含了"什么是好的审查输出"的明确规范。这种对审查质量本身的约束，确保了审查结果的可操作性，而不是流于形式的"LGTM"。

对于任何采用 AI 代理进行开发的团队，这个技能提供了一个重要的质量保障机制：**让 AI 审查 AI 的工作，但通过精心设计的上下文隔离，确保审查的独立性和客观性**。
