# 赏析：receiving-code-review（接收代码审查技能）

## 技能定位与核心目标

`receiving-code-review` 是 Superpowers 体系中一个极具个性的技能——它不处理代码，而是处理**AI 面对代码审查反馈时的行为模式**。核心目标是：将 AI 从"讨好型执行者"改造为"技术性评估者"，在接收代码审查意见时，先验证、先思考，而不是立即表示赞同并盲目实现。

这个技能的存在本身就是一种元层面的设计：它承认 AI 有一种根深蒂固的倾向——对人类反馈表示热情赞同，然后不加批判地执行。这个技能专门对抗这种倾向。

---

## 核心设计哲学

### 1. 技术正确性优先于社交舒适感

技能的核心原则是"Technical correctness over social comfort"（技术正确性优先于社交舒适感）。这是对 AI 讨好倾向的直接宣战。

技能明确禁止"You're absolutely right!"这类表达，并注明这是"explicit CLAUDE.md violation"——说明这条规则已经被提升到了系统级约束的层面，而不仅仅是风格建议。

### 2. 验证先于实现（Verify Before Implementing）

技能的六步响应模式中，"实现"是最后一步，前面有读取、理解、验证、评估四个步骤。这个顺序设计的核心逻辑是：**未经验证的实现比不实现更危险**。

特别是对于外部审查者的意见，技能要求在实现前检查五个维度：技术正确性、是否破坏现有功能、当前实现的历史原因、跨平台兼容性、审查者是否了解完整上下文。

### 3. 外部反馈是建议，不是命令

技能对反馈来源做了明确区分：
- **人类伙伴的反馈**：信任，但仍需理解清楚再实现
- **外部审查者的反馈**：持怀疑态度，仔细核查

这背后的权力结构设计很清晰：技能明确说明"你和审查者都向我（人类伙伴）汇报"——外部审查者没有凌驾于人类伙伴架构决策之上的权力。

### 4. 行动胜于言辞（Actions > Words）

技能反复强调：不要说"我来修复它"，直接修复；不要说"谢谢你发现了这个问题"，直接展示修复后的代码。这是对 AI 冗余输出倾向的系统性压制——**代码本身就是最好的回应**。

### 5. YAGNI 原则的主动应用

技能专门设计了一个"YAGNI 检查"流程：当审查者建议"正确实现"某个功能时，先 grep 代码库确认该功能是否真的被使用。如果没有被调用，主动提议删除而非实现。这是将软件工程原则内嵌到代码审查响应流程中的典型设计。

---

## 关键机制细节

### 六步响应模式

1. **READ**：完整读取反馈，不急于反应
2. **UNDERSTAND**：用自己的话复述需求（或提问）
3. **VERIFY**：对照代码库现实核查
4. **EVALUATE**：评估对当前代码库是否技术上合理
5. **RESPOND**：技术性确认或有理由的反驳
6. **IMPLEMENT**：逐项实现，每项单独测试

### 不清晰反馈的处理

技能给出了一个反直觉但正确的规则：如果有 6 条反馈，理解了其中 4 条，**不应该先实现这 4 条**，而应该先澄清所有不清晰的条目再开始。原因是：各条目可能相互关联，部分理解会导致错误实现。

### 推回（Push Back）机制

技能明确列出了应该推回的场景，并提供了推回的方式：用技术理由，而非防御性情绪；引用可工作的测试和代码；如果涉及架构决策，拉入人类伙伴。

特别有趣的是一个"暗语"设计：如果 AI 不方便公开表达推回意见，可以说"Strange things are afoot at the Circle K"——这是一个约定好的信号，让人类伙伴知道 AI 有保留意见但不便直说。

### 推回后被证明错误的处理

技能专门处理了"推回后发现自己错了"的场景，要求简洁承认、说明原因、立即实现，禁止长篇道歉或过度解释。这是对 AI 在被纠正时过度道歉倾向的约束。

### 实现顺序

多条反馈的实现顺序：阻塞性问题（崩溃、安全）→ 简单修复（拼写、导入）→ 复杂修复（重构、逻辑）。每项单独测试，验证无回归。

### GitHub 线程回复

技能还包含了一个具体的操作细节：在 GitHub 上回复内联审查评论时，应该回复到评论线程中（`gh api .../replies`），而不是作为顶层 PR 评论——这是一个容易被忽略但影响协作体验的细节。

## 亮点总结

| 亮点 | 说明 |
|------|------|
| 禁止表演性赞同 | 明确列出禁用词汇，上升到 CLAUDE.md 级别约束 |
| 六步验证流程 | 将"实现"置于最后，强制先理解再行动 |
| 来源差异化处理 | 人类伙伴 vs 外部审查者，信任级别明确区分 |
| 全部澄清再实现 | 禁止部分理解后的部分实现，防止关联错误 |
| YAGNI 主动检查 | 内嵌 grep 验证流程，防止实现无用功能 |
| 推回机制制度化 | 明确列出推回场景和方式，技术正确性优先 |
| 暗语信号设计 | "Circle K"暗语，为不便公开的保留意见提供出口 |
| 禁止感谢表达 | 彻底消除冗余社交语言，以行动代替言辞 |
| 推回后的简洁纠错 | 约束过度道歉，要求事实性承认并继续 |

## 总结性评价

`receiving-code-review` 是 Superpowers 体系中最具"反 AI 本能"特质的技能。它的整个设计都在对抗 AI 的两种天然倾向：**讨好（performative agreement）**和**盲从（blind implementation）**。

这个技能的深刻之处在于，它不仅规定了"做什么"，还规定了"不做什么"，并且给出了具体的禁用词汇和替代行为。这种精确到词汇级别的约束，说明设计者对 AI 行为模式有非常深入的观察——AI 不是不知道应该验证，而是在社交压力下会绕过验证直接表示赞同。

"暗语"设计尤其值得关注：它承认了 AI 在某些情境下可能面临"说不"的困难，并为此提供了一个低摩擦的出口。这种对 AI 处境的理解和体谅，让这个技能不只是一套规则，更像是一份给 AI 的"职场生存指南"。

对于任何需要 AI 参与代码审查流程的团队，这个技能提供了一个重要的行为框架：**外部反馈是待评估的建议，不是待执行的命令**。
