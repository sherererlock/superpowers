# 赏析：dispatching-parallel-agents（并行 Agent 调度技能）

## 技能定位与核心目标

`dispatching-parallel-agents` 是一个专为 AI 编程助手（如 Claude Code）设计的**并行任务调度技能**。其核心目标是：当面对两个或两个以上**相互独立、无共享状态**的任务时，将它们分发给多个专注的子 Agent 并发处理，从而大幅缩短总体解决时间。

这个技能本质上是在 AI 工作流中引入了"并发编程"的思维——就像多线程处理独立任务一样，让多个 Agent 同时工作，而不是让单一 Agent 串行地逐一解决问题。

---

## 核心设计哲学

### 1. 隔离上下文，精准投喂

技能的第一原则是：**子 Agent 绝不继承主 Agent 的会话历史或上下文**。主 Agent 必须为每个子 Agent 精心构造"刚好够用"的上下文——不多也不少。

这一设计有两层深意：
- **子 Agent 的专注度**：上下文越干净，Agent 越不容易被无关信息干扰，越能聚焦于自己的问题域。
- **主 Agent 的上下文保护**：主 Agent 将协调工作保留在自己的上下文中，不被子任务的细节污染，保持全局视野。

这与软件工程中的"关注点分离"（Separation of Concerns）原则高度一致。

### 2. 独立性是并行的前提，而非可选项

技能通过一棵决策树（DOT 图）明确了并行调度的适用边界：只有当多个问题**真正独立**（修复一个不影响另一个）且**可以并发**（无共享状态冲突）时，才应该并行调度。否则，应退化为串行或合并处理。

这种"先判断再行动"的设计，避免了盲目并行带来的冲突和混乱。

### 3. 输出契约化

每个子 Agent 的任务描述都必须明确"期望输出是什么"。这不是细节，而是协调的基础——主 Agent 在整合结果时，需要知道每个子 Agent 做了什么、改了什么，才能判断是否有冲突、是否需要进一步处理。

## 关键机制细节

### 决策树：何时并行？

```
多个失败？
  └─ 是 → 它们独立吗？
              ├─ 否（相关）→ 单 Agent 统一处理
              └─ 是 → 可以并发吗？
                          ├─ 是 → 并行调度
                          └─ 否（共享状态）→ 串行 Agent
```

这棵决策树是技能的"入口守卫"，防止在不适合的场景下滥用并行。

### Agent Prompt 的四要素

技能给出了高质量 Agent Prompt 的结构模板，包含四个核心要素：

| 要素 | 作用 |
|------|------|
| **Focused（聚焦）** | 明确单一问题域，防止 Agent 发散 |
| **Self-contained（自包含）** | 提供理解问题所需的全部上下文 |
| **Specific about output（输出明确）** | 规定 Agent 应返回什么 |
| **Constraints（约束）** | 限制 Agent 的修改范围，防止越界 |

### 常见错误对比

技能用"❌ 错误 vs ✅ 正确"的对比格式，直接点出四类高频失误：

- **范围太宽**：`"Fix all the tests"` → 应改为 `"Fix agent-tool-abort.test.ts"`
- **缺乏上下文**：`"Fix the race condition"` → 应粘贴具体错误信息和测试名称
- **无约束**：Agent 可能重构整个代码库 → 应加 `"Do NOT change production code"`
- **输出模糊**：`"Fix it"` → 应要求 `"Return summary of root cause and changes"`

### 整合阶段的验证清单

并行不是终点，整合才是关键。技能明确了四步验证流程：
1. 阅读每个 Agent 的摘要，理解变更内容
2. 检查是否有冲突（多个 Agent 是否编辑了同一文件）
3. 运行完整测试套件，验证所有修复协同工作
4. 抽查——Agent 可能犯系统性错误

## 亮点总结

| 亮点 | 说明 |
|------|------|
| **决策树驱动** | 用 DOT 图形式化"何时并行"的判断逻辑，避免主观随意 |
| **上下文隔离原则** | 子 Agent 不继承主 Agent 历史，保持双方上下文纯净 |
| **Prompt 四要素模板** | 给出可复用的高质量 Agent 指令结构 |
| **反模式明确列举** | 用对比格式直接展示常见错误，降低误用风险 |
| **真实案例佐证** | 以 2025-10-03 的真实调试会话为例，展示 3 Agent 并行解决 6 个失败的完整流程 |
| **整合验证清单** | 并行结束后的四步验证，确保结果可信 |

---

## 总结性评价

`dispatching-parallel-agents` 是一个**思维框架型技能**，它的价值不在于提供某种神奇的技术手段，而在于将"并发编程"的工程直觉迁移到 AI Agent 工作流中，并配套了完整的判断标准、操作模板和验证流程。

它最大的贡献是**把"什么时候该并行"这个隐性判断显式化**——通过决策树、反模式列举和真实案例，让使用者在面对多任务场景时有据可依，而不是凭感觉行事。

对于需要频繁处理大型代码库中多处独立故障的工程师或 AI 助手来说，这个技能能够将调试时间从"N 个问题串行"压缩到接近"1 个问题的时间"，是提升 AI 辅助开发效率的实用利器。
