# execute-plan — 从命令到技能的迁移路标

## 技能定位与核心目标

`execute-plan` 是一个已废弃的命令，指向替代方案 `superpowers:executing-plans` 技能。它的存在意义不在于执行任何计划，而在于**在旧接口和新接口之间架设一座临时的桥梁**，确保使用旧命令的用户能够被平滑引导到新的技能体系。

从命名上看，`execute-plan`（动词+名词的命令式）到 `executing-plans`（动名词+复数的技能式）的转变，本身就反映了 Superpowers 体系从"命令驱动"向"技能驱动"的架构演进方向。

## 核心设计哲学

### 命令式接口向技能式接口的系统性迁移

`execute-plan` 和 `brainstorm` 同属一批被废弃的命令，它们共同揭示了 Superpowers 体系的一次架构层面的转型：早期版本使用斜杠命令（`/execute-plan`、`/brainstorm`）作为用户与功能之间的接口，而新版本将这些功能重新实现为技能（skill），通过 `Skill` 工具按需加载。

这个转型的动机是清晰的：命令是静态的、预定义的，而技能是动态的、可组合的。技能可以被其他技能引用、可以按需加载、可以携带完整的上下文和指令，而命令只能触发一段固定的提示词。随着 Superpowers 体系的功能扩展，技能架构比命令架构具有更好的可扩展性。

### 废弃期作为用户教育窗口

保留废弃命令而非直接删除，本质上是把废弃期当作一个**用户教育窗口**。在这个窗口期内，每一次用户调用旧命令，都是一次向他们介绍新架构的机会。

对于 `execute-plan` 这个命令来说，这个教育意义尤为重要——它的替代品 `executing-plans` 不只是功能等价物，而是一个完整的计划执行方法论（包含任务分解、子代理协作、验证循环等）。废弃通知不只是说"换个名字"，而是在引导用户进入一个更丰富的工作流体系。

## 关键机制细节

### 统一的废弃模板：一致性优于个性化

`execute-plan` 与 `brainstorm` 的废弃实现几乎完全相同：相同的 description 格式、相同的正文结构、相同的措辞模式。这种一致性是刻意的——废弃通知不需要个性化，它需要的是**可预期性**。

用户在遇到第一个废弃命令时学会了模式（"这个命令废弃了，用对应的技能替代"），之后遇到其他废弃命令时就能立即理解，不需要重新解读。统一的模板降低了认知负担，让废弃迁移成为一个可重复的、低摩擦的过程。

### 命令名到技能名的显式映射

废弃通知中明确写出了替代技能的名称："superpowers executing-plans"。这个显式映射看似多余（用户可以自己猜），但在实践中非常重要：命令名（`execute-plan`，单数）和技能名（`executing-plans`，复数动名词）并不是简单的一一对应，用户无法通过命名规律自动推断。

显式写出替代方案的名称，消除了用户在迁移过程中可能遇到的最后一个摩擦点：不知道该用什么替代。
