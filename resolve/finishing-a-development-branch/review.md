# 赏析：finishing-a-development-branch（完成开发分支技能）

## 技能定位与核心目标

`finishing-a-development-branch` 是 Superpowers 开发工作流的**收尾引擎**，专门处理"实现完成后该怎么办"这个关键决策点。它的核心目标是：在代码实现完毕后，通过结构化的验证和选项呈现，引导开发者安全、清晰地完成分支的最终处置——无论是合并、提 PR、保留还是丢弃。

在整个 Superpowers 流水线中，它是最后一棒：
- **上游**：`executing-plans` 或 `subagent-driven-development` 完成实现后调用本技能
- **本技能**：验证 → 决策 → 执行 → 清理
- **配套**：`using-git-worktrees` 创建的隔离工作区由本技能负责清理

---

## 核心设计哲学

### 1. 测试验证是硬性前置条件

技能的第一步是运行测试套件，且**测试失败时直接停止，不进入后续流程**。这是一条不可绕过的硬性规则，而非建议。

背后的逻辑很简单：**带着失败测试合并代码，比不合并更糟糕**。这个设计将"质量门控"内嵌到工作流本身，而不是依赖开发者的自觉性。

### 2. 四选一的结构化决策

技能要求 AI 精确呈现四个选项（合并到本地、推送并创建 PR、保留分支、丢弃），不多不少，不加解释。这种设计有两层用意：

- **消除歧义**：开放式问题（"你想怎么处理？"）会导致对话发散；四个固定选项让决策路径清晰可预期
- **覆盖完整场景**：四个选项恰好覆盖了分支处置的所有合理结果，没有遗漏

### 3. 差异化的工作区清理策略

技能对四个选项的工作区（worktree）处理方式做了精细区分：
- 选项 1（本地合并）和选项 4（丢弃）：清理工作区
- 选项 2（创建 PR）和选项 3（保留）：**保留工作区**

这背后的判断是：PR 创建后可能还需要修改，保留工作区是合理的；而本地合并和丢弃后工作区已无价值，应当清理。这种差异化处理避免了"一刀切清理"导致的数据丢失风险。

### 4. 丢弃操作的强制确认机制

选项 4（丢弃）要求用户输入精确的字符串 `discard` 才能执行，而不是简单的"是/否"确认。这是一种**摩擦力设计**——故意增加操作难度，防止误操作导致不可逆的数据丢失。

## 关键机制细节

### Step 1：测试验证

- 支持多种测试命令（npm test / cargo test / pytest / go test）
- 测试失败时展示具体失败信息，并明确说明"无法继续"
- 这是整个技能唯一的硬性阻断点

### Step 2：确定基础分支

- 通过 `git merge-base` 自动探测基础分支（main 或 master）
- 探测失败时主动询问用户，而非猜测
- 这一步为后续的合并和 PR 操作提供准确的目标分支

### Step 3：呈现选项

- 严格要求"不加解释"——选项本身已足够清晰
- 这是对 AI 冗余输出倾向的主动约束

### Step 4：执行选项

**选项 1（本地合并）的完整流程：**
切换到基础分支 → 拉取最新代码 → 合并特性分支 → **再次验证测试** → 删除特性分支

注意：合并后还要再跑一次测试，确保合并结果本身也是干净的。

**选项 2（创建 PR）的模板化输出：**
PR 描述使用固定模板（Summary + Test Plan），降低 AI 生成随意内容的风险，同时保证 PR 信息的一致性和可读性。

**选项 4（丢弃）的确认流程：**
在要求输入 `discard` 之前，先展示将被删除的具体内容（分支名、提交列表、工作区路径），让用户清楚知道自己在放弃什么。

### Step 5：清理工作区

- 通过 `git worktree list` 检查当前是否在工作区中
- 仅对选项 1 和 4 执行 `git worktree remove`
- 这一步与 `using-git-worktrees` 技能形成闭环

## 亮点总结

| 亮点 | 说明 |
|------|------|
| 测试硬性前置 | 测试失败直接停止，质量门控内嵌于流程 |
| 四选一结构化决策 | 消除歧义，覆盖所有合理处置场景 |
| 差异化工作区清理 | 根据选项智能决定是否清理，避免误删 |
| 丢弃强制确认 | 要求输入精确字符串，防止不可逆误操作 |
| 合并后二次验证 | 本地合并后再跑测试，确保合并结果干净 |
| PR 模板化输出 | 固定 PR 描述结构，保证信息一致性 |
| 透明性声明 | 开始时主动告知使用本技能，便于人类监督 |
| 常见错误清单 | 明确列出"永远不要"和"永远要"的行为边界 |

## 总结性评价

`finishing-a-development-branch` 是一个**以防御性设计为核心**的收尾技能。它的每一个设计决策都在回答同一个问题：**如何防止 AI 在最后关头搞砸？**

测试硬性前置防止带病合并，四选一结构防止决策发散，差异化清理防止误删工作区，强制确认防止误丢代码——这些设计共同构成了一道多层安全网。

更值得关注的是技能对"快捷路径"的刻意拒绝：它没有提供"自动选择最优方案"的功能，而是始终要求人类做最终决策。这种设计承认了一个现实：**在代码集成这个高风险操作上，人类的判断权不应被 AI 代劳**。

作为 Superpowers 流水线的终点站，这个技能以严谨的流程设计，确保了整个开发周期能够以可控、可审查的方式画上句号。
